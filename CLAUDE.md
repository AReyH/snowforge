# CLAUDE.md — snowcraft

> A Python toolkit that wraps the most common, tedious Snowflake operations into
> clean, tested, pip-installable functions. Think of it as the standard library
> that the `snowflake-connector-python` never shipped.

---

## Project Overview

**Package name:** `snowcraft`
**PyPI target:** `pip install snowcraft`
**Author:** Arturo Rey
**Python support:** 3.10+
**License:** MIT

### What this package does

Data engineers who work with Snowflake spend a disproportionate amount of time
rewriting the same boilerplate: MERGE statements for incremental loads, schema
diffs before deployments, cost queries to find runaway warehouses, SCD Type 2
logic, and so on. `snowcraft` codifies those patterns into a stable, typed,
well-tested API so teams can stop copy-pasting and start importing.

### What this package does NOT do

- It is not an ORM. It does not abstract SQL away.
- It is not a replacement for dbt, SQLMesh, or any transformation framework.
- It does not manage Snowflake infrastructure (use Terraform for that).
- It does not handle authentication beyond what `snowflake-connector-python` already provides.

## Module Specifications

### `snowcraft/connection.py`

Provides a thin wrapper around `snowflake.connector.connect()` that supports
both direct credential passing and environment variable resolution. Exposes a
context manager for safe connection lifecycle management.

**Public API:**
```python
SnowforgeConnection(
    account: str | None = None,   # falls back to SNOWFLAKE_ACCOUNT env var
    user: str | None = None,
    password: str | None = None,
    database: str | None = None,
    schema: str | None = None,
    warehouse: str | None = None,
    role: str | None = None,
)

# Usage
with SnowforgeConnection() as conn:
    conn.execute("SELECT CURRENT_USER()")
```

**Implementation notes:**
- Use `__enter__` / `__exit__` for connection lifecycle.
- Validate that required params are present at instantiation time, not at execute time.
- Never log credentials, even at DEBUG level.
- Expose a `.cursor()` method that returns a typed cursor wrapper.

### `snowcraft/merge.py`

The centerpiece module. Generates and executes MERGE INTO statements for
incremental loads, abstracting away the boilerplate of match conditions,
update clauses, and insert clauses.

**Public API:**
```python
MergeBuilder(
    conn: SnowforgeConnection,
    target_table: str,              # "DATABASE.SCHEMA.TABLE" fully qualified
    source_query: str,              # Any SELECT statement
    match_keys: list[str],          # Columns that uniquely identify a row
    strategy: Literal["upsert", "append", "delete_insert"] = "upsert",
    update_columns: list[str] | None = None,  # None = all non-key columns
    watermark_column: str | None = None,       # For incremental filtering
    watermark_table: str | None = None,        # Where to store the watermark
)

MergeBuilder.build() -> str           # Returns the SQL string (for inspection/logging)
MergeBuilder.execute() -> MergeResult # Executes and returns stats

@dataclass
class MergeResult:
    rows_inserted: int
    rows_updated: int
    rows_deleted: int
    execution_time_ms: int
    query_id: str
```

**Implementation notes:**
- Use `sqlglot` to build the MERGE statement programmatically, not f-strings.
  This prevents SQL injection and handles identifier quoting correctly.
- The watermark logic should read the last known max value from `watermark_table`,
  inject it as a WHERE clause on `source_query`, then update the watermark after
  a successful merge. Wrap both in a transaction.
- Validate that `match_keys` exist in the source query columns before executing.
- The `build()` method must be callable without a live connection — useful for
  testing and for logging the SQL before execution.

**Example generated SQL (upsert strategy):**
```sql
MERGE INTO target_db.public.orders AS target
USING (
    SELECT * FROM source_db.public.orders_staging
    WHERE updated_at > '2024-01-01 00:00:00'
) AS source
ON target.order_id = source.order_id
WHEN MATCHED THEN UPDATE SET
    target.status = source.status,
    target.updated_at = source.updated_at
WHEN NOT MATCHED THEN INSERT (order_id, status, updated_at)
VALUES (source.order_id, source.status, source.updated_at);
```

### `snowcraft/schema.py`

Schema introspection and diffing. Given two fully-qualified table names (or
schemas), produce a structured diff that can be used in CI gates, alerting, or
migration planning.

**Public API:**
```python
SchemaInspector(conn: SnowforgeConnection)

SchemaInspector.get_columns(table: str) -> list[ColumnDef]
SchemaInspector.diff(source: str, target: str) -> SchemaDiff

@dataclass
class ColumnDef:
    name: str
    data_type: str
    is_nullable: bool
    default: str | None
    comment: str | None

@dataclass
class SchemaDiff:
    added: list[ColumnDef]       # In source, not in target
    removed: list[ColumnDef]     # In target, not in source
    type_changed: list[tuple[ColumnDef, ColumnDef]]   # (old, new)
    nullability_changed: list[tuple[ColumnDef, ColumnDef]]
    is_breaking: bool            # True if any removals or type narrowing

    def to_markdown(self) -> str: ...   # Human-readable diff table
    def to_dict(self) -> dict: ...      # For JSON serialization / CI output
```

**Implementation notes:**
- Query `INFORMATION_SCHEMA.COLUMNS` — do not use `SHOW COLUMNS` as it has
  different privilege requirements and returns inconsistent results.
- `is_breaking` should be `True` for: column removal, type change to a narrower
  type (e.g., VARCHAR(256) → VARCHAR(64)), changing nullable → not nullable.
- The `to_markdown()` output should be clean enough to post directly in a GitHub
  PR comment via a GitHub Actions step.

### `snowcraft/profiler.py`

Wraps Snowflake's `QUERY_HISTORY` and `WAREHOUSE_METERING_HISTORY` views to
surface expensive queries, slow patterns, and cost attribution.

**Public API:**
```python
QueryProfiler(conn: SnowforgeConnection)

QueryProfiler.top_expensive(
    n: int = 20,
    lookback_hours: int = 24,
    warehouse: str | None = None,
) -> list[QuerySummary]

QueryProfiler.find_full_scans(lookback_hours: int = 24) -> list[QuerySummary]

QueryProfiler.warehouse_cost(
    lookback_days: int = 7,
    group_by: Literal["warehouse", "user", "role"] = "warehouse",
) -> list[CostSummary]

@dataclass
class QuerySummary:
    query_id: str
    query_text: str
    user_name: str
    warehouse_name: str
    execution_time_ms: int
    bytes_scanned: int
    partitions_scanned: int
    partitions_total: int
    credits_used: float
    start_time: datetime
    optimization_hints: list[str]   # Generated heuristically

@dataclass
class CostSummary:
    group_key: str
    credits_used: float
    estimated_cost_usd: float       # Based on $3/credit default, configurable
    query_count: int
```

**Implementation notes:**
- `optimization_hints` should be generated heuristically based on query stats,
  not via LLM (keep this module dependency-free). Examples: "High partition scan
  ratio — consider a cluster key on this table", "Query returned 0 rows but
  scanned 500GB — check filter conditions".
- `find_full_scans` is defined as queries where `partitions_scanned / partitions_total > 0.8`
  and `bytes_scanned > 1GB`.
- Credits-to-USD conversion rate should be a configurable parameter with a
  sensible default, since Enterprise vs. Business Critical pricing differs.
- Always query `SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY` (not
  `INFORMATION_SCHEMA.QUERY_HISTORY`) for lookback > 7 days.

### `snowcraft/scd.py`

Slowly Changing Dimension helpers for Type 1 (overwrite) and Type 2 (versioned history).

**Public API:**
```python
SCDManager(
    conn: SnowforgeConnection,
    target_table: str,
    source_query: str,
    business_keys: list[str],
    tracked_columns: list[str],     # Columns that trigger a new SCD2 version
    effective_from_col: str = "effective_from",
    effective_to_col: str = "effective_to",
    current_flag_col: str = "is_current",
)

SCDManager.apply_type1() -> MergeResult
SCDManager.apply_type2() -> SCDResult

@dataclass
class SCDResult:
    rows_inserted: int       # New records
    rows_expired: int        # Old versions closed out
    rows_unchanged: int
    query_id: str
```

**Implementation notes:**
- Type 2 logic requires two operations in a transaction: (1) UPDATE existing
  current records to set `effective_to` and `is_current = FALSE` where the
  tracked columns have changed, (2) INSERT new versions with `effective_from =
  CURRENT_TIMESTAMP` and `is_current = TRUE`.
- Use Snowflake's multi-statement transaction support, not Python-level loops.
- The `effective_to` for active records should be stored as `9999-12-31` (the
  industry convention), not NULL, to simplify range queries.

## Coding Conventions

### General

- All public functions and classes must have Google-style docstrings.
- Type hints are mandatory on all function signatures. No `Any` unless
  genuinely unavoidable, and even then add a comment explaining why.
- Prefer `dataclasses` over plain dicts for structured return values — it makes
  the API self-documenting and IDE-friendly.
- Never use f-strings to build SQL. Always use `sqlglot` or parameterized
  queries. This is non-negotiable.

### Error handling

Raise from the custom exception hierarchy in `exceptions.py`:

```python
SnowforgeError           # Base
├── ConnectionError      # Can't connect / bad credentials
├── SchemaError          # Schema validation failures
├── MergeError           # MERGE execution failures
└── ProfilerError        # QUERY_HISTORY access issues (missing privilege)
```

Always include the original Snowflake exception as the `__cause__` when
re-raising (`raise SnowforgeError("...") from original_exception`).

### SQL style (for generated SQL)

- All Snowflake keywords in UPPERCASE.
- All identifiers double-quoted and fully qualified (database.schema.table).
- Generated SQL should be formatted with consistent indentation — it will be
  logged and humans will read it.

## Testing Strategy

### Unit tests (required for every PR)

Unit tests must mock the Snowflake cursor entirely using `pytest-mock`. They
test the logic of SQL generation, parameter validation, and result parsing
without touching a real warehouse.

```python
# Example pattern in tests/conftest.py
@pytest.fixture
def mock_conn(mocker):
    conn = mocker.MagicMock(spec=SnowforgeConnection)
    conn.execute.return_value = mocker.MagicMock()
    conn.execute.return_value.fetchall.return_value = []
    return conn
```

Every module must have >90% unit test coverage. Coverage is enforced in CI.

### Integration tests (run manually against a test account)

Integration tests live in `tests/integration/` and are skipped if the
`SNOWFLAKE_ACCOUNT` environment variable is not set. They use a dedicated
`SNOWFORGE_TEST` database that is created and torn down by the test session's
`conftest.py` fixtures.

Never use production credentials or production databases in tests.

## Release Process

1. Bump the version in `pyproject.toml` following SemVer:
   - PATCH for bug fixes and non-breaking improvements
   - MINOR for new public API additions
   - MAJOR for breaking changes to the public API
2. Update `CHANGELOG.md` with a summary of changes.
3. Commit with message: `chore: bump version to X.Y.Z`
4. Push a tag: `git tag vX.Y.Z && git push origin vX.Y.Z`
5. The `publish.yml` GitHub Action will build and push to PyPI automatically.

Never push directly to `main`. All changes go through a PR with at least one
passing CI run.

## Roadmap (Future Modules)

These are not yet implemented but are planned for future minor releases:

- `snowcraft/clone.py` — Zero-copy clone management for dev/test environment
  provisioning. Automate creating a cloned dev environment from prod, running
  tests, and tearing it down.
- `snowcraft/rbac.py` — Role and privilege auditing. Answer questions like
  "which roles have access to this table?" and "which users have ACCOUNTADMIN?"
  programmatically.
- `snowcraft/tasks.py` — Wrapper around Snowflake Tasks for defining and
  managing scheduled SQL with Python, as an alternative to Airflow for
  Snowflake-native orchestration.
- `snowcraft/streams.py` — Helpers for working with Snowflake Streams (CDC),
  including stream lag monitoring and consuming stream data into target tables.

## Key Design Principles

**Inspectability over magic.** Every operation that generates SQL must expose a
`build()` or `to_sql()` method so engineers can see exactly what will be
executed before running it. No surprises.

**Minimal dependencies.** The core package should have exactly two required
dependencies: `snowflake-connector-python` and `sqlglot`. Everything else is a
dev dependency. This keeps the install fast and avoids version conflicts in
downstream environments.

**Fail loudly and early.** Validate inputs at construction time, not at
execution time. If a `match_keys` column doesn't exist in the source query,
raise an error before touching the warehouse.

**Return structured objects, not raw tuples.** Every method that returns data
returns a dataclass, not a list of tuples. This makes the library pleasant to
use in IDEs and makes future API changes non-breaking.
